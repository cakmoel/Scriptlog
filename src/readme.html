Welcome to Scriptlog

Your personal blogware platform!

Scriptlog is a free and an open source PHP blog software. We're refactoring our legacy weblog code. Making it more simple, secure, modular, fast and robust personal blogging system. Scriptlog runs on PHP 7.2 or higher and uses MySQL.

Scriptlog's base original code was forked from Piluscart 1.4.1. We took her out from her weblog engine that support blog functionality on Piluscart 1.4.1. Obviously Scriptlog is Piluscart's little sister. She was being refactored to help her siblings going further on next generation of Piluscart development which is undefined yet.

Scriptlog is not a replacement for a full Content Management System. Do not plan on building the next CMS Framework with it! However, it is meticulously engineered for several reasons.

  • To power parts of weblog or personal website that did not need a full CMS.
  • Runs small scripts and systems without a lot of overhead.
  • Provides a base speed for your personal weblog day to day.
  • To provide a secure foundation for your blog.
  • Emphasizing simplicity, privacy and security for blog software.

Before you install Scriptlog on your host, make sure Scriptlog can work correctly in your environment. Scritplog need the following system requirement.

Requirements:

  • Apache with mod_rewrite enabled (.htaccess files) or nginx (rewrite configuration covered in Documentation)
  • A minimum of 32MB of memory allocated to PHP
  • MySQL version 5.5 or greater or any version of MariaDB
  • PHP version 7.2 or greater is recommended
  • PDO
  • mysqli
  • mysqlnd
  • curl
  • Gd
  • iconv
  • fileinfo
  • mbstring
  • exif
  • filter_list
  • zip
  • HTTPS Support

Installation:

1) Download Scriptlog

Every release of Scriptlog is available for download, whether the current stable release, upcoming test-builds, or previous versions for historical intererst. When you visit the download page you will find a link to each of the two main editions.

Stable releases of Scriptlog are intended to be used by general public, so these would be the appropriate choice for the majority of users. The current Scriptlog release can always be downloaded from the Get Scriptlog page.
Older stable releases can also be downloaded from the Stable Release section of the Sourceforge website.

Test builds of Scriptlog are intended to be used by developers and testers to identify any bugs or issues during the development process. These are not suitable for use as a main blogging system by the general public. However, should you wish to try out a test build(preverably in local webhost), each current release of the Test Builds can be downloaded on Scriptlog's repository.

2) Ensure you have the required system.

if you have not the required system yet, Scriptlog may not work properly in your host environment.

3) Unzip the package downloaded

Extract it in an empty directory. You will see the following files, folders or directories after you extract it

  • admin/
  • install/
  • lib/
  • public/
  • config.sample.php
  • .htaccess
  • index.php
  • license.txt
  • readme.html

4) Upload Scriptlog

Upload all files, folders or directories above through FTP/SFTP or whatever upload method you prefer to the public-facing directory of your site.

5) Ensure that permission for :

install, public/log, public/cache, public/themes and library/plugins folders are set to writeable and that all files belong to the web user or is a part of the same group as the web user.

6) Create database for Scriptlog

You may name it anything you like. Choose utf8mb4_general_ci as it collation. The method for database creation varies depending on your webhost but may require using adminer, PHPMyAdmin or etc. If you are unsure how to createt this, ask your host.

7) Follow the installation procedure

Open install/ in your browser. You will be asked to fill each column provided. You should enter all database connection details and administrator account. These informations will help you through process to set up a config.php file.

if some reason this does not work, don't worry. Open up config.sample.php with text editor and change the following line:

return [ 'db' => [ 'host' => 'hostname', # database's hostname 'user' => 'username', # database's username 'pass' => 'password', # database's password 'name' => 'dbname' # database's name ], 'app' => [ 'url' => 'http://yourservername', # your server-name 'email' => 'email address', # administrator's email 'key' => 'application key' # optional ] ];

fill in your database connection details, your server name and administrator's email.
Save the file as config.php and upload it.

Open install/ in your browser. Once the configuration file is set up, the installer will set up the tables needed for your blog. If there is an error, double-check your config.php file, and try again.

8) Remove the installation directory

For security purpose, delete the install folder when you are finish.

Usage

After you finish installation, you can populate your blog. To manage your blog content, go to administrator panel on web browser: http://your-server-name/admin/.

How the System Works

When a request is sent to the folder where Scriptlog resides, the .htaccess file will send the request to index.php as front-controller. The index will load the main engine files in "library" and then checks to see if a "cached" version of the page already exists. If not, then it will continue to load the core system files.

After this, it will check with dispatcher to figure out what route and rules to call.

  1. If a route requested mattching the URL is found in rules, then the method given (either in the URL or config) is run and the output is returned.
  2. If a route requested is not found, then it will trigger a 404 error as the output.

The output (whether an error or not) is then placed in the current theme and the final content is sent to the users browser

Contributing

Everyone is welcome to contribute code to the project. There are several ways to help out:

Prerequisites

All contributors need to have an account at Github

How To

  1. Create an issue on Github if you have found a bug.

  3. Write patches for open bug/feature issues, preferably with test cases included

Bug fix workflow

Either submit a ticket or choose a ticket you wish to contribute to. Fix the bug in a topic branch and make a pull request.
Sometimes bug fixes are trivial enough to not require a ticket, but a detailed commit message is enough.
